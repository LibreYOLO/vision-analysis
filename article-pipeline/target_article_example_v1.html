Model comparison, LibreYOLO benchmark set

YOLOv9-S vs YOLOX-S

Two small detectors at nearly identical compute (13.52 vs 13.46 GFLOPs). YOLOv9-S is more accurate. YOLOX-S is faster. This article shows by how much, where the gap comes from in the pipeline, and whether the ranking survives different runtimes and hardware.

COCO val2017 5000 images input 640 batch 1 PyTorch FP32 RTX 5080 primary
Headline. At batch 1 PyTorch FP32 on an RTX 5080: YOLOv9-S scores 46.8 mAP@50-95 vs 40.5 for YOLOX-S (+6.3 points, +15.56% relative). YOLOX-S returns 188.18 fps_mean vs 167.09 (+12.63% relative, 0.67 ms per frame saved) and uses 191 MB less peak VRAM. The ranking holds on A100, DGX Spark, RPi5, and under ONNX FP32 and TensorRT FP16.

They are not comparable to mAP and are not used to rank the models. They appear once in the full table below, flagged, and are not discussed further.

Accuracy

YOLOv9-S is higher on every mAP axis reported by the harness. The gap widens at a tighter IoU threshold (+3.9 at mAP@50 vs +7.3 at mAP@75), which reflects better localization. The gap is largest in absolute terms on large objects (+9.86), and largest in relative terms on small objects (+21.6% relative, from 17.80 to 21.65).

These numbers come from LibreYOLO's retrained checkpoints (provider: libreyolo 0.2.0, weightsOrigin: retrained). Other weight sources (original author releases, independent retrains) can yield different values. Anyone comparing to numbers from a different release should re-evaluate against the same harness.

Latency and throughput

YOLOX-S is 12.63% faster end to end (188.18 vs 167.09 fps_mean on the RTX 5080). The breakdown per frame:

  • Inference dominates the gap. 4.1343 vs 4.7912 ms is 0.66 ms of the 0.67 ms total gap. Preprocess and postprocess roughly cancel: YOLOX-S is 0.08 ms faster at preprocess and 0.07 ms slower at postprocess (NMS).
  • Both models show low jitter. fps_mean and fps_p50 agree to within 1% on both: YOLOv9-S 167.09 mean vs 165.49 p50, YOLOX-S 188.18 mean vs 187.33 p50. The JSONs do not report p95 or standard deviation, so tail behavior needs its own measurement.
  • GFLOPs are effectively identical (13.52 vs 13.46) yet the throughput gap is 12.63%. Theoretical compute does not predict wall clock here. Measure, don't estimate.

Memory

YOLOv9-S has fewer parameters (7.23M vs 8.97M) but uses more peak VRAM at inference (1313 vs 1122 MB). Parameter count is weight storage; peak VRAM includes activations, workspace buffers, and postprocessing state. For this pair, YOLOv9-S costs 191 MB more on GPU despite being the smaller model on disk. Peak host RAM is near equal (2546 vs 2526 MB).

Both models fit comfortably on a 16 GB RTX 5080 at batch 1. If you co-locate several models on a single GPU, or plan larger batches, the 17% VRAM gap matters. At batch 1 on this card it does not.

Does the ranking survive other runtimes and hardware?

Yes. Same evaluation, same input size, same dataset. The columns below come from the matching benchmark JSONs in the set.

Accuracy is stable: ONNX FP32 produces the same mAP as PyTorch FP32; TensorRT FP16 drops mAP@50-95 by 0.2 points on both models. The throughput ranking is preserved; on RTX 5080 the YOLOX-S lead actually widens slightly under ONNX and TensorRT (12.6% to 15-17%), meaning the YOLOv9-S graph is marginally less friendly to those runtimes' kernel fusion paths. TensorRT FP16 peak VRAM is 684 MB for YOLOv9-S and 626 MB for YOLOX-S, roughly half the PyTorch FP32 footprint.

TensorRT FP16 is a 2.8x throughput multiplier for both models. 463.28 vs 167.09 for YOLOv9-S, 535.69 vs 188.18 for YOLOX-S. If you deploy on an RTX 5080 and do not convert, the PyTorch-vs-TensorRT decision matters more than the model choice. The model ranking is the same either way.

When to pick which

  • Pick YOLOv9-S when accuracy is the product KPI and 5-6 ms per frame fits your latency budget. +6.3 mAP@50-95 points is a substantial gap at this model size, and it holds across all hardware and runtime combinations measured here. MIT licensing also matters to some teams.
  • Pick YOLOX-S when throughput is binding, or when VRAM budget is tight (multiple models on one GPU, large batches, co-resident workloads). 12-20% more throughput and ~17% less peak VRAM across the board.
  • If you will deploy on an NVIDIA GPU: convert to TensorRT FP16 first. 2.8x throughput with a 0.2 mAP drop is a larger lever than the model choice itself.

What this data does not tell you

  • Batched throughput. All timings are batch 1. Batching changes the GPU utilization picture, and the ranking can compress or invert at higher batch sizes. Re-measure at your target batch.
  • Tail latency. Only fps_mean and fps_p50 are reported. p95 and p99 are not, and mean/median agreement within 1% does not guarantee a tight tail.
  • INT8. No INT8 numbers in this set. If PTQ or QAT is in your plan, benchmark both models in your target runtime before deciding.
  • NMS and confidence thresholds. The benchmark harness does not record the confidence threshold used for the single-point precision and recall fields, or the NMS IoU. Use mAP for ranking; generate your own PR curves for shipping operating points.
  • Energy and sustained load. 5000 images warms the card but does not exercise a thermal ceiling. No wall-power data is recorded.
  • Your data. These are COCO numbers. On domain data (aerial, medical, industrial, crowded retail), the gap may shrink, grow, or reverse. Fine-tune and re-evaluate before you pick.

Try it with LibreYOLO

Both models ship in LibreYOLO. Install once, then point at the weights file for either family. LibreYOLO auto-detects which architecture to load from the filename prefix (LibreYOLO9 for YOLOv9, LibreYOLOX for YOLOX). 

# pip install libreyolo
from libreyolo import LibreYOLO

# YOLOv9-S: 7.23M params, 46.8 mAP@50-95, MIT
model = LibreYOLO("LibreYOLO9s.pt")
results = model("image.jpg")

# YOLOX-S: 8.97M params, 40.5 mAP@50-95, Apache-2.0
model = LibreYOLO("LibreYOLOXs.pt")
results = model("image.jpg")

To reproduce the exact benchmark numbers in this article, run the validation harness against COCO val2017 with input 640, batch 1, PyTorch FP32. The LibreYOLO9 and LibreYOLOX classes work the same way if you prefer explicit construction. Full docs at libreyolo.com/docs.
